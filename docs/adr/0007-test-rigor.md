# 7. Test rigor as the safety net for a churning dependency

Status: accepted (2026-06-19)

## Context

Two decisions raise the cost of a weak test suite to unacceptable:
- ADR-0001 tracks GoMLX `main`, which is mid-refactor (`ml/context`â†’`ml/model`,
  `optimizers`â†’`optimizer`). The vendor API *will* break under us.
- ADR-0006 bumps Go and all deps often. Every bump is a chance for silent
  numerical or behavioral regression.

On a stable stack, tests catch your own bugs. Here they must also catch the
*dependency's* changes â€” and do it loudly, in one place, the moment a bump lands.

## Decision

Treat tests as more important than usual, and make the green gate the merge bar
for every change including dependency bumps (ADR-0006). Specifically:

- **Backend boundary contract tests (the early-warning system).** Pin the
  observable behavior of the `backend` interface (ADR-0002): given inputs, the
  matmul/gradient/optimizer-step outputs are asserted numerically. When a GoMLX
  `main` bump changes vendor behavior, these fail in the `backend` module **only**,
  not as mysterious downstream loss divergence. This is the single most valuable
  test set in the repo.
- **High-coverage unit tests on owned deps.** The pure-Go libs (BPE tokenizer,
  safetensors, GGUF, dataloader) are deterministic logic with no vendor coupling â€”
  test them thoroughly, including edge cases (empty/odd-length inputs, dtype
  boundaries, content-hash split determinism, round-trip encode/decode).
- **Layer-parity tests (model).** Each block (RMSNorm, RoPE, attention, SwiGLU)
  asserts its forward output matches a committed PyTorch golden fixture within
  tight tolerance (1e-4 bf16 / 1e-6 fp32). Per DESIGN.md.
- **Numerical correctness.** Gradients vs finite-difference; the overfit-a-batch
  test driving loss â†’ ~0 before any full run.
- **Integration tests across module boundaries.** backendâ†”modelâ†”train wired
  end-to-end on a tiny synthetic problem, not just per-unit.
- **Structural tests.** The backend-boundary grep test (ADR-0002), run per module.
- **Tokenizer equivalence gate** (ADR-0003): the pure-Go BPE vs HF on the real
  corpus; CI fails on any mismatch.

Golden fixtures (PyTorch reference tensors, tokenizer expected outputs) are
committed so tests are reproducible without a Python runtime in CI.

## Consequences

- A dependency bump that changes numerics cannot merge silently â€” a boundary
  contract test goes red first, in the module that owns the vendor.
- Writing the boundary contract tests is front-loaded work before leaning on the
  adapter; this is deliberate, not overhead.
- Per CLAUDE.md: a bug is reproduced with a failing test first, and "done" means
  the real path was run and observed â€” green units alone are not "done" for the
  CPU/CUDA bring-up runs.
- Some fixtures are large (reference tensors); they live in-repo for reproducibility
  and are regenerated by a committed, documented Python script when a block changes.
